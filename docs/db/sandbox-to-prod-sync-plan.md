# Sandbox ‚Üí Production schema sync (database-1 ‚Üí database-2)

## Context

Two AWS RDS Postgres instances in `ap-south-1`:
- **Sandbox** (`database-1.c9w88wq6qyco‚Ä¶`) ‚Äî driven by `.env.local` / Infisical `dev`. Source of truth.
- **Production** (`database-2.c9w88wq6qyco‚Ä¶`) ‚Äî driven by `.env.production` / Infisical `prod`. Effectively empty.

The Drizzle schema at `src/lib/db/schema.ts` (86 `pgTable` definitions, 2861 lines, `leads` alone is ~592 lines) was renamed to snake_case in commit `10af73a` (2026‚Äë04‚Äë29, on `main`). The last drift snapshot at `docs/db/drift/2026-04-26-2132/summary.json` predates the rename and reports:
- diff3 (sandbox‚Üîprod) ‚Äî 13 only‚Äësandbox, 3 only‚Äëprod, 13 shape diffs
- diff4 (code‚Üîprod) ‚Äî 3 only‚Äëcode, 4 only‚Äëprod, 66 shape diffs

The differences are too large to chase column‚Äëby‚Äëcolumn, and prod has no transactional data to lose. Decision: **regenerate the diff, wipe `public` in prod, push the current `src/lib/db/schema.ts` schema, then clone a small allowlist of reference rows from sandbox**.

## Phase 3 ‚Äî Wipe and rebuild

The repo wraps DB tooling in `infisical run --` (see `package.json:6-32`). The convention for prod ops is `infisical run --env=prod -- <cmd>` (mirroring `test:e2e:prod-eval`). Either that or exporting `DATABASE_URL` from `.env.production` works.

```bash
# 0. Pin a timestamp for the run
TS=$(date -u +%Y-%m-%d-%H%M)
mkdir -p backups

# 1. Insurance backup of prod (even if "empty")
PROD_URL=$(grep '^DATABASE_URL=' .env.production | cut -d= -f2- | tr -d '"')
pg_dump --no-owner --no-privileges "$PROD_URL" > backups/prod-pre-wipe-$TS.sql

# 2. Wipe prod's public schema
psql "$PROD_URL" -v ON_ERROR_STOP=1 -c "DROP SCHEMA public CASCADE; CREATE SCHEMA public; GRANT ALL ON SCHEMA public TO postgres; GRANT ALL ON SCHEMA public TO public;"

# 3. Push the current Drizzle schema into prod
#    --strict makes drizzle-kit prompt before applying.
infisical run --env=prod -- npx drizzle-kit push --strict

# 4. Clone Bucket A rows from sandbox to prod, in one pg_dump invocation
#    so FK order is preserved (product_categories before products).
SANDBOX_URL=$(grep '^DATABASE_URL=' .env.local | cut -d= -f2- | tr -d '"')
pg_dump --data-only --column-inserts \
  -t product_categories \
  -t products \
  -t risk_hypotheses \
  -t nbfc_tenants \
  "$SANDBOX_URL" \
  | psql "$PROD_URL" -v ON_ERROR_STOP=1 -1
```

Notes on the commands above:

- AWS RDS Postgres has no Supabase `auth.*` schema (the app migrated off Supabase per `scripts/migrate-supabase-to-aws.mjs`). `DROP SCHEMA public CASCADE` is therefore safe ‚Äî there is nothing else to preserve. `src/lib/db/schema.ts` does not declare any `pgSchema(...)` (verified ‚Äî zero matches), so `public` is the only schema.
- `drizzle-kit push` is the right tool because the migration journal at `drizzle/meta/_journal.json` only tracks 3 of the on‚Äëdisk SQL files; `drizzle-kit migrate` would be unreliable. `drizzle/0033_align_prod_with_sandbox.sql` and `drizzle/0034_full_sandbox_to_prod_sync.sql` are obsolete additive patches and should not be reapplied after the wipe.
- `--column-inserts` is robust against later column reorders. `pg_dump` with multiple `-t` flags emits tables in dependency order, which handles the `products ‚Üí product_categories` FK without a manual ordering step.
- `psql -1` wraps the data load in a single transaction so a row failure rolls everything back.
- All Bucket A tables have `uuid().defaultRandom()` PKs, so there are no sequences to `setval()` after the load.

If the user later approves cloning a table that has serial/integer sequence PKs (e.g. `kyc_data_audit`), append `SELECT setval(pg_get_serial_sequence('<table>', 'id'), MAX(id)) FROM <table>;` on the prod side after the load.

## Phase 4 ‚Äî Verify

```bash
# Expect: diff3 == "No differences" for every Bucket A/B table's shape;
# diff4 == "No differences".
npm run db:drift
cat docs/db/drift/<new-ts>/summary.json

# Refresh the per-table catalog (now reflects post-rebuild prod row counts).
npm run db:catalog

# Spot-check row counts in prod.
psql "$PROD_URL" -c "SELECT relname, n_live_tup FROM pg_stat_user_tables ORDER BY n_live_tup DESC, relname LIMIT 30;"
```

Pass criteria:

1. `summary.diff3.summary` shows zero shape diffs and zero only‚Äëin‚ÄëX tables.
2. `summary.diff4.summary` shows zero shape diffs and zero only‚Äëin‚ÄëX tables.
3. Bucket A row counts in prod match sandbox; every other table is `0`.
4. App boots: `infisical run --env=prod -- npm run build && infisical run --env=prod -- npm run start` (or the existing PM2 `ecosystem.prod.config.js`). Hit `/login` ‚Äî no `relation ‚Ä¶ does not exist` errors.

There is no `drizzle-kit push --dry-run` flag; the post‚Äëpush `db:drift` is the verification.

## Risks & guardrails

- **`scripts/migrate-supabase-to-aws.mjs:15-23`** has plaintext DB credentials in git. Out of scope for this task but should be rotated and removed; flagging because it's adjacent.
- **`drizzle-kit push --strict` is interactive.** The user must be at the terminal to confirm. Drop `--strict` for CI/scripted runs only after a dry run with `--strict --verbose` once.
- **Row cloning is irreversible against the freshly‚Äëpushed prod schema.** If a table is mis‚Äëbucketed (e.g. `nbfc_tenants` row is actually test data), the recovery is to truncate that one table ‚Äî cheap because no other prod data references it yet.
- **Backup file contains DB credentials (in the connection comments) and possibly PII**: `backups/` is already gitignored; keep it that way and don't commit `backups/prod-pre-wipe-*.sql`.
