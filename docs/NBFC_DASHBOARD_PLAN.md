# NBFC Risk Dashboard — Architecture & Plan

> Status: Phase A + B in progress (scaffold + LLM). Phases C and D queued for follow-up sessions.
> Branch: `feature/nbfc-dashboard` off `apoorv-claude`

## What this is

A multi-tenant operational dashboard for NBFC partners who finance EV vehicles fleet-managed
through Itarang. Each NBFC sees only its own borrowers' loans + the IoT telemetry from
those borrowers' vehicles. The Risk page surfaces up to 20 cards per severity bucket
(High Alert / Warning / OK) generated by a LangGraph agent that:

1. **Plans** which risk dimensions to investigate (asset health, behavioural,
   geographical, financial, seasonal).
2. **Proposes hypotheses** as testable statements (e.g. "borrowers with ≥40 % 7-day
   km drop have ≥3× higher 30+ DPD probability").
3. **Fetches data** through tool wrappers around CRM (loans, KYC) and IoT
   (telemetry on the VPS) read-only Postgres connections.
4. **Runs the test** (initially in-process JS analytics; later sandboxed Python).
5. **Critiques** the result against statistical sanity checks and the existing
   risk-thresholds.ts.
6. **Scores severity** (High / Warning / OK) and populates evidence_json.
7. **Writes the card** to `risk_card_runs`. Up to 60 cards (3 buckets × 20)
   per tenant, refreshed every 4 h or on-demand.

## Data model additions (Phase A)

All new tables live in the existing CRM Supabase Postgres. Tenant scoping is enforced
in application code (drizzle `where` clauses) rather than RLS for now — RLS comes in
Phase C when we wire NBFC partner auth.

### `nbfc_tenants`

One row per NBFC partner.

```ts
nbfc_tenants(
  id            uuid PK,
  slug          text unique,         // 'demo-nbfc', 'sample-nbfc'
  display_name  text,
  contact_email text,
  aum_inr       decimal,             // tenant AUM for header pill
  active_loans  integer,             // denormalized count, refreshed nightly
  created_at    timestamptz,
  updated_at    timestamptz
)
```

### `nbfc_users`

Links a CRM `users.id` to one or more `nbfc_tenants`. Most users belong to exactly one
tenant; some operators (Itarang internal) may belong to many.

```ts
nbfc_users(
  user_id    uuid FK -> users.id,
  tenant_id uuid FK -> nbfc_tenants.id,
  role       varchar(32),           // 'admin','viewer'
  created_at timestamptz,
  PRIMARY KEY (user_id, tenant_id)
)
```

### `nbfc_loans`

Bridges existing `loan_applications` (and via `lead_id` chain, the borrower + vehicle)
to a tenant. One loan belongs to one NBFC.

```ts
nbfc_loans(
  loan_application_id varchar(255) PK FK -> loan_applications.id,
  tenant_id            uuid FK -> nbfc_tenants.id,
  vehicleno            varchar(64),  // joins to IoT vehicle_state.vehicleno
  emi_amount           decimal,
  emi_due_date_dom     smallint,    // day-of-month
  current_dpd          integer,     // refreshed nightly
  outstanding_amount   decimal,
  created_at           timestamptz,
  updated_at           timestamptz
)
```

### `risk_hypotheses`

The hypothesis catalogue. Each hypothesis is a stable identity (so cards can be tracked
over time). Initially seeded with 5 hand-coded entries; LangGraph proposes new ones.

```ts
risk_hypotheses(
  id               uuid PK,
  slug             text unique,         // 'usage-drop-7d', 'dpd-7-and-low-utilization'
  title            text,
  description      text,                // 1-paragraph statement of the hypothesis
  test_method      text,                // 'sql' | 'python' | 'js'
  test_definition  jsonb,               // SQL / Python source / JS predicate AST
  source           varchar(16),         // 'human' | 'llm-v1'
  created_at       timestamptz,
  retired_at       timestamptz          // soft-delete
)
```

### `risk_card_runs`

One row per (tenant, hypothesis, run_date). The Risk page reads the latest run per
(tenant, hypothesis).

```ts
risk_card_runs(
  id                uuid PK,
  tenant_id         uuid FK,
  hypothesis_id     uuid FK,
  run_at            timestamptz,
  severity          varchar(16),       // 'high' | 'warn' | 'ok'
  finding_summary   text,              // 1-line headline
  affected_count    integer,
  total_count       integer,
  evidence_json     jsonb,              // sample rows, distribution stats, chart spec
  llm_critique      text,
  llm_model         text,               // 'gpt-4o' | 'gpt-4o-mini' | 'gpt-5'
  llm_prompt_tokens integer,
  llm_completion_tokens integer
)
```

## IoT bridge — connection details

The CRM gets read-only Postgres access to the VPS via a second drizzle client.

- **Connection**: `IOT_DATABASE_URL=postgres://dashboard_ro:<pw>@72.61.246.37:5433/intellicar?sslmode=require`
- **Client**: `src/lib/db/iot.ts` exports `iotDb` plus typed query helpers. Schema is
  *not* mirrored in drizzle (we don't run migrations against the IoT DB) — queries are
  written as `sql` template tags returning typed result rows.
- **Security setup** lives in `phase5_pg_external.sh` on the VPS:
  - Bind Postgres container to `0.0.0.0:5433`
  - `pg_hba.conf` allows `host all dashboard_ro 0.0.0.0/0 scram-sha-256` with `hostssl` only
  - `dashboard_ro` is GRANTed only on telemetry tables (`vehicle_state`, `telemetry_gps`,
    `telemetry_battery`, `telemetry_can`, `alerts`, `vehicles`, `daily_distance_per_vehicle`)
  - Self-signed SSL cert generated and Postgres `ssl=on`
  - Firewall (ufw / Hostinger) opens 5433 inbound

## LangGraph workflow

`src/lib/ai/langgraph/risk-hypothesis-graph.ts`. State machine:

```
START
  ↓
plan_dimensions          ── decides which categories to investigate (asset/behavioural/geo/financial)
  ↓
propose_hypotheses       ── LLM proposes 3-5 testable statements per dimension
  ↓
fetch_data (per hyp)     ── tools: getCrmLoanSlice, getIotFleetSummary, getIotVehicleHistory
  ↓
run_test (per hyp)       ── currently JS predicate or SQL; sandboxed Python in Phase D
  ↓
critique (per hyp)       ── sanity-check: enough sample size? statistical signal vs. noise? confounders?
  ↓
score_severity (per hyp) ── high/warn/ok via thresholds + critique confidence
  ↓
write_card (per hyp)     ── upsert risk_card_runs
  ↓
END
```

Tools the agent can call (typed, server-side only):

| Tool name                | What it returns                                             |
|--------------------------|-------------------------------------------------------------|
| `getCrmLoanSlice`        | aggregate of nbfc_loans filtered by tenant_id + DPD/EMI cuts |
| `getIotFleetSummary`     | counts of online/offline/with_lat/avg_soc for a vehicleno set|
| `getIotVehicleHistory`   | 24h GPS+CAN slice for a single vehicleno                    |
| `getCohortBaseline`      | last-90-day distribution of a metric for the tenant         |
| `getThresholds`          | the existing risk-thresholds config                          |

Cost guardrails:
- One full run uses ~10-20 LLM calls. Default model: `gpt-4o-mini` ($0.15/1M in, $0.60/1M out)
- Hard ceiling: 50k completion tokens per run, fail-fast on exceed
- Cache plan + propose stages for 24h per tenant (don't re-plan on every refresh)

## How to run after Phase A

1. **Enable VPS PG external access** (one-time):
   ```bash
   scp phase5_pg_external.sh root@72.61.246.37:/opt/intellicar/storage/
   ssh root@72.61.246.37 'cd /opt/intellicar/storage && bash phase5_pg_external.sh'
   ```
   Save the printed `IOT_DATABASE_URL` somewhere safe.

2. **Update CRM env** (Vercel + local `.env.local`):
   ```env
   OPENAI_API_KEY=sk-...
   IOT_DATABASE_URL=postgres://dashboard_ro:...@72.61.246.37:5433/intellicar?sslmode=require
   ```

3. **Push the schema**:
   ```bash
   npm run db:push
   ```

4. **Seed a demo NBFC tenant + sample loan links**:
   ```bash
   tsx scripts/seed-nbfc-demo.ts
   ```

5. **Manually grant yourself the role**:
   ```sql
   UPDATE users SET role = 'nbfc_partner' WHERE email = 'your-test@example.com';
   INSERT INTO nbfc_users (user_id, tenant_id, role) VALUES (...);
   ```

6. **Visit** `/nbfc/risk` → see the 5 hand-coded cards rendering live data
   from the IoT bridge + your CRM loans table. Click "Re-run analysis" to
   exercise the LangGraph workflow (will take 30-60s).

## Open questions / risks

- **No NBFC users in production yet** — Phase C must build the signup/invite flow before
  this is usable by external NBFC partners.
- **IoT VNO ↔ borrower mapping** is approximated through `nbfc_loans.vehicleno`. We don't
  yet have a robust source of truth that links a financed loan to a specific vehicleno.
  Need to clarify which CRM table stores this mapping.
- **Direct internet PG exposure** is a real attack surface. Mitigated by TLS + read-only
  + scram-sha-256 + strong password rotation, but Cloudflare Tunnel or Tailscale would be
  stronger. Worth revisiting before going live with paying NBFC tenants.
- **Cost cap on LangGraph** — each tenant refresh costs ~$0.05-0.20 in LLM tokens. If we
  ship to N partners and auto-refresh every 4h that's 6 × N × $0.10/day. Manageable but
  worth metering per tenant.
